# MSTアルゴリズム Python 実装 3種比較実験仕様書

## 1. 文書情報

* 文書名: MSTアルゴリズム Python 実装 3種比較実験仕様書
* 版数: 0.2
* 作成日: 2026-04-13
* 目的: Python で利用可能な MST 系合成データ実装 3 種を、同一条件下で比較し、有用性・安全性・処理性能・導入性の観点から採用判断に資する知見を得る

---

## 2. 背景

MST（Maximum Spanning Tree）は、差分プライバシー付きの表形式合成データ生成手法として広く参照される手法の一つであり、カテゴリデータの marginal 保持に強いとされる実装群が存在する。OpenDP SmartNoise のドキュメントでは、MST はカテゴリデータに対して高い性能を示す手法として紹介され、`snsynth` から利用できる形で公開されている。([SmartNoise][1])

一方で、同じ「MST」と呼ばれる実装でも、以下が異なる可能性がある。

* 前処理の有無と内容
* ドメイン定義方法
* ライブラリの抽象度
* 差分プライバシー機構の扱い
* 実験向け実装か、実務導入向けラッパか

そのため、「アルゴリズム名が同じ」であることをもって実装同士を同等と見なすのではなく、**同一入力・同一条件・同一評価指標で横並び比較**する必要がある。

---

## 3. 実験の目的

本実験の目的は以下のとおり。

1. MST の Python 実装 3 種について、有用性・安全性・処理時間・導入容易性の差異を明らかにする
2. 将来の評価基盤として採用すべき実装を選定する
3. Claude Code に比較実験の実装、評価、可視化、考察生成を依頼できる粒度まで仕様を具体化する
4. 今後の PrivBayes、AIM 等との比較にも拡張可能な実験枠組みを整備する

---

## 4. 比較対象

本実験では、`uv` により個別環境を作成し、`pip install` 可能な以下 3 実装を対象とする。

### 4.1 A. smartnoise-synth

* パッケージ名: `smartnoise-synth`
* 導入方法: `pip install smartnoise-synth`
* 位置づけ:

  * 高レベル API
  * `snsynth` 経由で `Synthesizer.create("mst", ...)` により MST を呼び出せる
  * SmartNoise ドキュメントでは、MST は Private-PGM を利用すると記載されている。([SmartNoise][1])

### 4.2 B. tmlt.private_pgm

* パッケージ名: `tmlt.private_pgm`
* 導入方法: `pip install tmlt.private_pgm`
* 位置づけ:

  * グラフィカルモデルベースの DP 推定・推論ライブラリ
  * PyPI 上で `private-pgm` 系の fork として公開されており、MST を含む複数メカニズムが実装されている。([PyPI][2])
  * 研究用コードであり、本番用途には注意が必要と明示されている。([PyPI][2])

### 4.3 C. dpmm

* パッケージ名: `dpmm`
* 導入方法: `pip install dpmm`
* 位置づけ:

  * Python 3.10 / 3.11 を前提とする DP 合成表データ生成ライブラリ
  * PyPI で公開されており、MST / PrivBayes / AIM を提供すると記載されている。([PyPI][3])
  * 研究・探索用途を想定し、複雑な本番用途向けではない旨が記載されている。([PyPI][3])

---

## 5. 比較の基本方針

比較の公平性を確保するため、以下を原則とする。

1. **データは共通**
2. **前処理ルールは可能な限り共通**
3. **epsilon 等の主要パラメータは揃える**
4. **生成件数は揃える**
5. **乱数 seed を固定する**
6. **各条件は複数回実行し、平均とばらつきを取得する**
7. **結果の良し悪しは単一指標で決めず、複数観点で総合判断する**

## 7. 実験データ

### 7.1 対象データの要件

MST はカテゴリデータとの親和性が高い。そのため、最初の比較は **単表・カテゴリ主体の表データ** を対象とする。

対象データの条件:

* 1 行 = 1 レコード
* 複数列のカテゴリ属性を含む
* 連続値は必要に応じて離散化
* 欠損値の扱いを明示
* ラベル列があれば下流タスク評価に利用可能

### 7.2 推奨データ構成

まずは 1 データセットで比較を成立させる。その後、必要に応じて 2〜3 データセットへ拡張する。

候補:

* 公開カテゴリデータセット
* 既存の検証用単表データ
* 属性列が多すぎず、列数 10〜30 程度の表

### 7.3 前処理ルール

前処理は比較結果に大きく影響するため、ルールを固定する。

#### 共通前処理

1. 不要列削除
2. 列型定義
3. 欠損値補完または欠損カテゴリ化
4. 連続値のビニング
5. 高カーディナリティ列の処理
6. カテゴリ値の整数エンコード
7. train / holdout 分割

#### 原則

* 各ライブラリの自動前処理に委ねすぎない
* 可能な限り **前処理済みの共通入力** を渡す
* ドメイン（列ごとのカテゴリ数）は共通管理する

## 8. 比較条件

### 8.1 epsilon

以下の候補で比較する。

* 0.1
* 0.3
* 1.0
* 3.0
* 10.0

### 8.2 生成件数

* 原則として元データ件数と同数
* 追加確認として、0.5倍 / 1倍 / 2倍 の生成件数も検討可能

### 8.3 繰り返し回数

* 各条件につき 5 回以上
* 平均、標準偏差、最良値、最悪値を記録

### 8.4 タイムアウト

* fit: 30 分
* sample: 10 分

超過時は失敗として記録する。

## 9. 評価観点

本実験では、以下 5 観点で比較する。

### 9.1 有用性

合成データが元データの統計的特徴や分析価値をどの程度保持するか。

### 9.2 安全性

合成データから元データのレコードや属性をどの程度推測できるか。

### 9.3 処理性能

学習・生成に必要な時間、メモリ、安定性。

### 9.4 実装容易性

環境構築、API の分かりやすさ、コード量、エラー対応のしやすさ。

### 9.5 将来拡張性

今後の比較基盤としての使いやすさ。PrivBayes、AIM、複数データセットへの横展開可能性。

---

## 10. 有用性評価指標

### 10.1 単変量分布

各列ごとに元データと合成データの分布差を比較する。

候補指標:

* TVD
* Jensen-Shannon Divergence
* KL Divergence（必要に応じて）
* 分布一致率

### 10.2 二変量分布

列ペアごとのクロス集計の一致度を比較する。

候補指標:

* 平均 TVD
* 最大 TVD
* 代表列ペアのクロス集計差

### 10.3 依存構造保持

MST の本質に近いため重要。

候補指標:

* 相互情報量の差
* Cramér's V の差
* 列ペア依存構造の順位相関

### 10.4 下流タスク有用性

特定の列を目的変数とし、合成データで学習したモデルの性能を元データ holdout で評価する。

候補:

* accuracy
* macro F1
* ROC-AUC
* PR-AUC

### 10.5 多様性・カバレッジ

合成データの多様性を確認する。

候補:

* ユニークレコード率
* 重複率
* 希少カテゴリ再現率
* カテゴリカバレッジ

---

## 11. 安全性評価指標

### 11.1 最近傍距離系

元データと合成データの近さを測る。

候補:

* DCR
* NNDR
* 平均最近傍距離
* 最小最近傍距離

### 11.2 完全一致・準一致リスク

* 完全一致件数
* 完全一致率
* 準一致率（距離閾値あり）

### 11.3 メンバーシップ推定

少なくとも 1 種の簡易 MIA を実施する。

候補:

* 最近傍ベース MIA
* holdout 区別ベース MIA

評価指標:

* ROC-AUC
* attack advantage
* precision / recall

### 11.4 属性推定

既知属性から未知属性を当てられるかを見る。

候補:

* 属性推定精度
* ベースラインとの差分

### 11.5 高 epsilon 時の劣化確認

epsilon 増加に伴う安全性低下が急かどうかを観察する。

## 14. 比較結果の解釈方針

以下の誤解を避ける。

1. 有用性が高い = 良い、とは限らない
   → 安全性とのトレードオフがある

2. 安全性が高い = 実務採用、とは限らない
   → 有用性や処理時間が不足する可能性がある

3. 実装が簡単 = 評価基盤として優れる、とは限らない
   → 将来の比較拡張性や透明性も重要

### 採用判断の基本軸

* **短期の PoC 向け**
* **研究・改造向け**
* **継続評価基盤向け**

の 3 観点で整理する。

## 21. リスクと留意事項

### 21.1 最大の比較バイアス

ドメイン定義と前処理の違い。
これを揃えないと比較結果は実装差でなく前処理差になる。

### 21.2 private_pgm の性格

研究用コード寄りであり、本番適用の安全性まで含意しない。PyPI 説明にも、浮動小数点近似ノイズ機構に関する注意が記載されている。([PyPI][2])

### 21.3 dpmm の位置づけ

比較的新しいライブラリで、MST 以外も扱える。MST の勝敗だけでなく、将来の比較基盤候補として見るべきである。([PyPI][3])

### 21.4 SmartNoise の位置づけ

使いやすさは高いが、内部的には Private-PGM を利用するため、独立した理論系というより「実務向けラッパ」として評価するのが妥当。([SmartNoise][1])
